# Alembic Migrations

Rules for database migrations using Alembic (the standard SQLAlchemy migration tool).

---

## Core Rules

- **Always autogenerate migrations** via your project's Alembic-autogenerate command — typically a Makefile target (`make generate-migration`), a `just` recipe, or `alembic revision --autogenerate -m "..."` directly. Never write migration files by hand.
- **Never manually edit migration files** in `alembic/versions/` — they are build artifacts.
- **Always review autogenerated migrations** before applying — verify column types, constraints, and table names match your intent.
- **One migration per logical change** — don't combine unrelated schema changes.
- **Never alter schema directly** in production.

## How Migrations Run

- **Locally:** the database (Postgres) runs in **Docker** (typically via `docker-compose up -d`). Alembic itself runs **on your host machine** (via `poetry run alembic upgrade head` or a Makefile target like `make apply-migration`) and connects to the Dockerized DB on `localhost:5432`. So: Docker hosts the database container, your host hosts the alembic process.
- **Production:** run via CI/CD pipeline, with `DATABASE_URL` set to the production database.
- **`alembic/env.py`** reads `DATABASE_URL` from the OS environment — no fallbacks, fails fast if missing.

## Alembic Configuration

Two files Alembic depends on:

- **`alembic.ini`** — top-level Alembic config (sets file locations, naming templates). Has a `sqlalchemy.url` field but leave it as a dummy placeholder — the real DB URL comes from `env.py` at runtime.
- **`alembic/env.py`** — Alembic's runtime script that the `alembic` CLI executes on every migration command. Two responsibilities:
  - **Read `DATABASE_URL`** via `os.environ["DATABASE_URL"]` — no fallback.
  - **Register all ORM models** via one import side-effect: `import src.persistence.models` (substitute your package path). The import populates SQLAlchemy's `Base.metadata` so autogenerate can detect changes.
